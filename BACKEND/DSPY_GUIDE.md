# Declarative Architecture and Optimization Guide: DSPy + Gemini 2.5 Flash

This document establishes best practices and architectural patterns for building AI applications using the DSPy framework integrated with the **Google Gemini 2.5 Flash** model. The main focus is to avoid unnecessary complexity and manual prompt engineering, ensuring high-quality responses through programmatic configuration and strong typing, without the need for a prior training dataset.

## 1. Declarative Architecture Principles (Avoiding Complexity)

- **Abandon Manual Prompting:** Do not write long instruction strings or concatenate variables manually. DSPy is designed to focus on declarative programming. Instead of telling the model _how_ to do something, declare _what_ must go in and come out using **Signatures**.
- **Use Built-in _Zero-Shot_ Modules:** Since there is no data for training weights or selecting examples (_few-shot_), rely on the model's inherent reasoning capabilities. Replace the basic `dspy.Predict` module with **`dspy.ChainOfThought`** for complex tasks. It forces the model to generate a logical reasoning step (`rationale`) before the final response, which drastically increases accuracy without adding complexity to the codebase.

## 2. Gemini 2.5 Flash Configuration and "Thinking Capabilities"

The Gemini 2.5 Flash model supports hybrid deep reasoning features that can be activated via code to improve the LLM's logical thinking.

- **Connectivity:** Use the `dspy.LM` class with the OpenAI-compatible endpoint via LiteLLM (`openai/gemini-2.5-flash`) or native access (`gemini/gemini-2.5-flash`).
- **Thinking Budget:** Gemini 2.5 Flash has the capacity for internal deliberation before responding. For tasks requiring high logical precision, configure the `thinking_budget` (or `reasoning_effort` mapped by LiteLLM) parameter at request time.
- **Cost/Latency Management:** Increasing the thinking budget improves the response but impacts latency and cost. Use higher budgets only in specific pipeline modules that truly require advanced logic.

## 3. Prompt Optimization via "Signatures" and "Schemas"

The best way to optimize Gemini's response in a scenario without a dataset is through the strict definition of its Signature:

- **Docstring as System Prompt:** DSPy maps your Signature class's `docstring` directly as the global System Prompt for the LLM. Be clear, concise, and define the expected semantic role.
- **Use Pydantic for Structured Outputs:** In extraction or classification tasks, use the native integration with **Pydantic** (`dspy.TypedPredictor` or directly declared schemas). Gemini 2.5 has excellent API-level decoding support for _JSON Schemas_, ensuring the response strictly follows the defined schema of keys, types, and order.
- **Field Descriptions (`desc`):** To improve model adherence, use the `desc` argument in `dspy.InputField()` and `dspy.OutputField()` declarations. This adds explicit semantic hints into the final prompt generated by the compiler. Variable names should be simple and semantic (e.g., `context`, `question`, `summary`) so that DSPy formats them properly.

## 4. Choosing the Ideal Adapter

DSPy uses adapters to translate its Signatures into API messages. The choice of adapter is crucial when working with Gemini 2.5 Flash:

- **JSONAdapter (Latency and Structure Optimization):** Use the `JSONAdapter` for strictly structured responses. It generates a payload without redundant formatting (like text markers), resulting in faster responses with fewer _tokens_, leveraging the Gemini API's native format coercion.
- **ChatAdapter (Tool Use / Web Search):** The `JSONAdapter` might fail if you are invoking tools (_tool calling_) integrated with Gemini (such as Google Search). If your architecture anticipates tool usage (e.g., the `dspy.ReAct` architecture), stick with the **`ChatAdapter`** (DSPy's default), which structures inputs using easily parsable text delimiters (like `[[ ## field ## ]]`).

## 5. Runtime Quality and Auto-Correction (Backtracking)

Since you won't be running long cycles of weight optimization or example selection (_few-shot_), quality will be guaranteed at runtime using DSPy's **Dynamic Backtracking** mechanism.

- **Logical Validation:** Create Python functions that validate business rules on the LLM's output (e.g., "does the generated text contain fewer than 280 characters?", "is the JSON valid?").
- **Use of Suggestions (`dspy.Suggest`):** Use suggestion statements to monitor and police these validations. When a suggestion fails, DSPy automatically halts execution, injects the previous incorrect output (_Past Output_) alongside a corrective instruction provided by you (_Instruction_), and triggers the model again to self-correct (momentarily increasing the temperature).
- **Avoiding Halts:** Prefer `dspy.Suggest` over `dspy.Assert`. `Suggest` will attempt to fix errors up to a pre-defined retry limit. If the model fails to conform, the system logs the failure but returns the best-effort result found without crashing or halting the application (`AssertionError`).
